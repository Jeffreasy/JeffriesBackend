## Rationale
The user's stated focus is AI + Telegram, and that is the right place to spend most of the next two months — but two things outrank it on pure risk math and are cheap enough that deferring them is indefensible: there is no provable DB restore path for single-instance Postgres holding bank transactions, loonstroken, and LaventeCare invoices/encrypted client credentials; and production secrets (bunq = real money movement, Google = owner-token minting) sit in a chat transcript. Both fixes are mostly dashboard actions a solo dev can do in an afternoon, with near-zero maintenance burden — exactly the kind of work that should happen first because it is high-impact, low-effort, and one-time. The big test/infra items from the review docs (handler-layer tests, render.yaml, schema consolidation) are real but are slow-burn hardening that compete directly with the AI/Telegram roadmap for the same scarce solo-dev hours; they belong woven in opportunistically, not as a blocking phase. RAG stays parked — it needs an external embedding-provider decision (xAI/Groq have none) and is a big bet, not a next step. The decisive framing for a solo maintainer: close the irreversible risks cheaply now, then pour the bulk of the 1-2 months into the AI/Telegram surface the user cares about, and treat reliability debt as small recurring side-quests rather than a stop-the-world project.

## Per-angle investigations

### Reliability & operability for a solo dev: what keeps this two-binary Go system up and recoverable, ranked by impact/effort, grounded in the actual code (not the now-stale 06-19 REVIEW.md).
First, what the REVIEW.md backlog gets WRONG today because it predates this engagement: CI now exists (.github/workflows/ci.yml runs go build/vet/test on every push+PR), the config.Validate boot-guard is live (cmd/api/main.go:30, refuses change-me secret in prod), and tests have grown to ~1572 lines across 20 files (config, engine, store, middleware, ai, google, mail, server all have suites that pass). The sync_runs observability we added is genuinely good: recordingCron wraps every Google sync (cron_workers.go:685), Record/Recent/DeleteOlderThan with a 14-day prune cron, surfaced in /sync/status. So several "open" reliability items are effectively closed.

What is STILL open and actually threatens uptime/recovery, in order:

(1) The HTTP handler layer — 8,220 lines, ~110 routes including all bank (transactions, loonstrook) and LaventeCare care/billing write paths — has literally 0.0% test coverage (go test -cover: internal/handler = "no test files"). This is the single biggest place a test would catch the most, because it's the widest blast radius and the least covered. The engine/store/middleware are partially covered (8-39%); handlers are a black hole.

(2) Schema dual-source is still live and still a fresh-boot footgun. runtime_schema.go's EnsureRuntimeSchema does ALTER TABLE on notes, personal_events, brain_preferences, email_sync_meta, lc_leads, lc_projects, lc_action_items and FKs devices — none of which it CREATEs. Those tables exist ONLY in migrations/*.sql, which no code runs. migrations/ has been kept in lockstep up to 026 (so it's NOT dead-and-forgotten — it's dead-but-load-bearing), meaning a restore-to-fresh-DB or DR rebuild boot-loops until someone hand-runs 22 SQL files. This is a recovery-time problem, not a steady-state one.

(3) No DB backups / PITR anywhere (no pg_dump job, no render.yaml, only a REVIEW.md mention). Bank + care + invoice data on a single Render Postgres with no provable restore path. Combined with (2), recovery from data loss is currently undefined.

(4) Observability blind spots that remain after sync_runs: cronTelegramHealthAlert (cron_workers.go:639) only nags about pending-action confirmations. Nothing proactively alerts on (a) bridge death — the local engine dying means WiZ lights silently stop; bridge_heartbeat.last_seen is written and shown in /focus + /settings but never triggers a Telegram alert; (b) a sync-failure STREAK — sync_runs records ok=false but only Google invalid_grant (wrapGoogleCron) and pending-calendar dead-letters fire an alert; a Gmail/schedule/Todoist job failing every tick is WARN-logged and otherwise invisible; (c) the api process itself — /health exists but no external uptime monitor pings it, so a Render crash-loop is silent.

(5) Deploy is dashboard-only (no render.yaml) and the Dockerfile builder uses floating golang:alpine — config-as-no-code plus non-reproducible builds; both raise recovery time if the service is mis-set or deleted.

Net: the system is steadily-up and well-instrumented for the happy path; its real exposure is RECOVERY (no backups + un-runnable schema) and PROACTIVE FAILURE DETECTION (bridge/sync/process death is observable in-app but never pushed). Those, plus the untested handler layer, are where the next effort pays off.

### SECURITY — secret-rotation blast radius, the single shared X-API-Key authz model, the bunq money-movement surface, and input-handling gaps, grounded in backend/internal + REVIEW.md.
The core auth hardening from the earlier engagement is real and present in the code: constant-time X-API-Key compare (server.go:177), a boot guard that refuses weak APP_SECRET_KEY outside development (config.go:192-203), un-spoofable rate-limit keying with TRUSTED_PROXY_COUNT defaulting to 0 (ratelimit.go:62-86), and a CORS allow-list that no longer reflects arbitrary origins (server.go:148-167). The bunq money-movement surface is genuinely well-defended: every payment request goes through a two-step pending-confirmation queue (laventecare.go:645-725), the executor re-fetches the invoice and refuses if ProviderRequestID/PaymentURL already set (executor.go:2767-2773), amounts are validated > 0, and the introspect/settings endpoints only ever return masked tails and booleans (settings.go:877-895) — never raw keys or full IBANs. LaventeCare customer secrets are AES-GCM encrypted and WRITE-ONLY: there is no decrypt function anywhere and the list endpoint returns only `secret_value_encrypted IS NOT NULL` as a boolean. No secret is written to logs.

The pasted-secret incident is the live issue. Rotation priority by blast radius: bunq API key (full production banking account — money movement + introspection) and GOOGLE_CLIENT_SECRET (mints Gmail/Calendar tokens for the owner with the existing refresh token) are the two that genuinely demand immediate rotation; MICROSOFT_CLIENT_SECRET (Graph app-permission mail-send) and TELEGRAM_BOT_TOKEN (bot drives the 75-tool AI executor, including /approve on the bunq confirmation flow) are close behind; the Clerk secret is contained to the Vercel app by the single-tenant model; Grok/Groq keys are billing-abuse only. APP_SECRET_KEY was already rotated, but because BridgeAPIKey defaults to APP_SECRET_KEY (config.go:129) and the local engine sends it as X-API-Key to Render over the wire (cloud_bridge.go:229), that one secret still spans three trust boundaries (main API auth, bridge auth, and the LaventeCare credential-encryption KEK) — so the rotation should also split these into separate secrets.

Remaining gaps are all defense-in-depth, not open doors: the shared single X-API-Key with no per-scope separation; client-supplied userId on user-scoped routes (mitigated by single-tenant + restricted Clerk sign-up); the bunq HTTP-level idempotency hole (B4 — a lost-response retry can create a second RequestInquiry because X-Bunq-Client-Request-Id is regenerated per attempt); no request-body size limit anywhere (no MaxBytesReader in the codebase); and three configured-but-unverified webhook secrets (GAS, Telegram bridge, bunq callback) that gate no endpoint and are pure dead indicators in the settings UI.
