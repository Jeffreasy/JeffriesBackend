## Verdict
The notes feature is functionally rich and, critically, NOT a source of permanent data loss in the common single-surface flow — every body-changing update snapshots the prior version into note_revisions in-transaction and the editor exposes restore, so the concurrency bugs are recoverable. However, it ships an unsafe offline story: the app advertises a working "Offline modus actief" mode while offline writes are silently dropped with zero error surfaced. That single gap turns an everyday action (jot a note with no signal) into real, unrecoverable loss of note content. Combined with a privacy-masking bypass via search, the feature over-promises two of its most trust-sensitive guarantees (offline durability, privacy masking). Ship-blocking on those two; the rest are quality/correctness debt that is real but bounded and largely recoverable. Verdict: solid core, two trust-breaking defects that must be fixed before relying on offline or privacy mode.

**Biggest risk:** Silent loss of note content when creating/editing offline. providers.tsx configures no mutation defaults, so TanStack's default networkMode:'online' PAUSES the write while offline; mutateAsync never settles, so NoteEditor.handleSave hangs with saving=true and no error, the optimistic note looks saved, and because /notes is in PERSIST_DENY_PREFIXES nothing is dehydrated to IndexedDB and no paused mutation is persisted. Closing the PWA before reconnecting discards the note entirely. For a personal journaling/knowledge PWA the owner relies on daily, this is the worst outcome (unrecoverable note-data loss) on a plausible everyday trigger, made more dangerous because the UI explicitly tells the user offline mode is active. ## Themes
- Over-promised guarantees: the two highest-severity defects are both cases where the UI advertises a guarantee it does not deliver — a persistent 'Offline modus actief' banner over writes that are silently dropped, and a privacy-masking mode whose search still queries and counts plaintext. Trust-sensitive promises are silently false.
- No optimistic concurrency anywhere on the notes write path: the backend PATCH is a bare full-field overwrite (note.go) and multiple client surfaces (card checkbox toggle, the open editor) send full bodies from stale/frozen snapshots. A single server-side version guard would neutralize a whole class of last-writer-wins findings.
- Denormalized data without lifecycle maintenance: cached business_context_title and free-text business_context_id have no FK and aren't updated/nulled on CRM delete or rename, so reads (Telegram, editor) drift out of sync with the source of truth.
- Tag enrichment runs continuously and divergently from the visible UI: alias/hashtag detection over free-text body rewrites live tags on every keystroke and again at save time with a different result than what's shown, producing phantom dirty-state and stored tags the user never chose.
- UI state derived from stale or wrong inputs: empty-states, dirty-flags, and editor drafts are all computed from snapshots or flags that ignore the live/contextual state (viewMode, scope, privacyOn, refetched note), a recurring 'render-time snapshot vs current reality' gap.

## Notably solid
- Body-changing updates are recoverable: every inhoud overwrite snapshots the prior note into note_revisions inside the same transaction (note.go:177-194) and the editor exposes restore (onLoadRevisions/onRestoreRevision), which is exactly why the two concurrency findings were correctly downgraded from data-loss to recoverable medium.
- Deliberate, correct cache hygiene: /notes, /personal-events and /schedule are intentionally excluded from IndexedDB persistence (PERSIST_DENY_PREFIXES, providers.tsx:14), avoiding stale-write replay hazards — the offline gap is a missing fail-fast, not a wrong persistence choice.
- CRM deletion handles PII responsibly within its own domain: DeleteCompany cascades contacts and relies on FK-nulling for leads/projects/documents so no orphaned contact PII remains — the gap is only the cross-domain notes link, not the company data itself.
- Evidence of prior security/UX hardening that held up: an earlier fix already stopped live re-adding of free-text #hashtags to tags (NoteEditor.tsx:563-566 comment), and the remaining tag issues are the narrower alias/save-time residue, showing the area is being actively maintained.
- Optimistic-update UX and a real revision/restore safety net make the editor feel responsive and forgiving; the core single-surface note lifecycle (create, edit, autosave, restore) is sound.

### 🟠 HIGH (3)

**1. Offline note creates/edits are silently lost — paused mutations are never persisted or replayed despite an "offline mode" banner**  
- _Data layer, hooks, offline & sync (useNotes, IndexedDB persist, optimistic updates)_ · `offline-sync` · `C:/Users/jeffrey/Desktop/Projecten/JeffriesHomeapp/app/providers.tsx:22-34`
- **Impact:** The UI advertises a working offline mode (PwaRegistry renders a persistent "Offline modus actief" banner). When offline, a usePostNotes/usePatchNotesId mutation is PAUSED under TanStack's default networkMode:'online' — its mutateAsync promise never settles, so onError never fires (no rollback, no toast) and NoteEditor.handleSave stays stuck with saving=true, saveError empty, and never calls onClose. The optimistic note from onMutate stays on screen looking saved. Because /notes is in PERSIST_DENY_PREFIXES (providers.tsx:14) the notes query — including the optimistic insert — is NOT dehydrated to IndexedDB, and paused mutations are not persisted either. Closing the tab/PWA before reconnecting discards the create/edit entirely: real loss of note content (journaling + knowledge base) with zero error surfaced. For a personal journaling PWA this is a plausible everyday trigger.
- **Fix:** Either (a) make writes fail fast offline so the existing error UI engages — set defaultOptions.mutations.networkMode:'always' so offline mutations reject immediately, letting NoteEditor.handleSave set saveError and keep the content/quick-capture text; or (b) implement true offline support: persist the MutationCache (PersistQueryClientProvider can dehydrate mutations) with setMutationDefaults({ mutationFn }) so paused mutations rehydrate, and call queryClient.resumePausedMutations() on the window 'online' event in PwaRegistry. Option (a) is the minimal fix to stop silent loss; (b) matches the advertised offline mode. At minimum, gate the "Offline modus actief" banner messaging so users don't trust offline writes.

**2. Note search runs against hidden note content while privacy mode is on — content-existence oracle that defeats masking**  
- _Cross-cutting: backlink graph, search, privacy, business-context, symbols_ · `privacy` · `C:/Users/jeffrey/Desktop/Projecten/JeffriesHomeapp/app/notities/page.tsx:133`
- **Impact:** With notes privacy mode enabled (cards show '••••'), the search box stays fully functional, and the local content filter (titel+inhoud+tags) ignores privacyOn. Typing any term changes the visible-card count rendered as `${displayed.length} zichtbaar`, letting an onlooker confirm whether a note containing a given word/name/diagnosis/client exists and roughly how many — defeating the masking the owner explicitly enabled. The masked content is fully present in JS memory and queryable.
- **Fix:** When privacyOn is true, disable the search/tag filters (and the search input) or skip the local content filter so the visible set and count do not vary by query. At minimum, do not narrow `displayed` by note.inhoud/note.titel/tags while masked.

**3. Deleting (or renaming) a LaventeCare CRM company leaves notes pointing at a dangling/stale business_context_id and a stale cached business_context_title**  
- _Cross-cutting: backlink graph, search, privacy, business-context, symbols_ · `data-integrity` · `C:/Users/jeffrey/Desktop/Projecten/JeffriesBackend/backend/internal/store/laventecare.go:258`
