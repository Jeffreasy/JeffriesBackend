# Personal-data backends audit — 2026-06-22

_Read-only multi-agent audit (notes, habits, personal-events, schedule/rooster, cross-cutting), each finding adversarially verified. 24 confirmed._

**Totaal:** 🔴 1 · 🟠 6 hoog · 🟡 9 medium · ⚪ 8 laag

## Verdict
This backend is fundamentally sound in its data model and Amsterdam-timezone discipline, but it carries one latent defect that would silently destroy the schedule and personal-event data on any fresh or disaster-recovery rebuild, plus a cluster of correctness bugs that quietly corrupt the two features the owner depends on daily: shift-filtered habit streaks and the salary/contract hour totals. None of these will surface as a loud crash — they fail silently and produce plausible-but-wrong numbers, which is the worst failure mode for a personal-data app the owner trusts without double-checking. The biggest single risk is the ON CONFLICT / missing-unique-index defect (schedule.go:103 + runtime_schema_base.go:80,187): the runtime bootstrap (the ONLY schema path — migrations/ is dead code) creates schedule and personal_events WITHOUT the unique (user_id, event_id) index that every upsert's ON CONFLICT clause requires. The current production DB only works because it was built by the now-dead migrations. The moment this app is restored, redeployed to a fresh Render instance, or rebuilt for DR, the very first schedule sync throws Postgres 42P10, rolls back the whole transaction, and the schedule/personal-event tables never populate — after which the salary check silently reports 0 planned hours. The FreshDB test passes today only because it never exercises an actual upsert. Fix this first.

**Biggest risk:** The ON CONFLICT / missing-unique-index defect on schedule and personal_events (runtime_schema_base.go:80 & :187 vs the ON CONFLICT (user_id,event_id) upserts at schedule.go:103 and personal_event.go:158/179). Because the runtime bootstrap is the only schema path and migrations/ is dead, any fresh deploy, restore, or DR rebuild produces tables without the unique index the upserts require; the first schedule/personal-event sync throws Postgres 42P10, the whole transaction rolls back, the tables stay empty, and the salary/contract check then silently reports 0 planned hours. It is invisible today only because the live prod DB was built by the now-dead migrations and the FreshDB test never runs a real upsert.

## Top fixes
1. 🔴 **Add unique index (user_id,event_id) to schedule AND personal_events in ensureBaseTables; make FreshDB test exercise a real BulkUpsert/Upsert** — `backend/internal/store/runtime_schema_base.go:80,187 (consumers: schedule.go:103, personal_event.go:158,179)`
   - The runtime bootstrap is the only schema path (migrations/ is confirmed dead). Without the matching unique index, every ON CONFLICT upsert raises 42P10 and rolls back, so schedule + personal_events never populate on any fresh/restored/DR DB — exactly the path the FreshDB test was meant to protect but doesn't, because it only smoke-tests reads. This is silent total data loss for the roster on the DR path, with downstream 0-hour salary reporting.
2. 🟠 **All-day work shifts get Duur=0 and silently undercount salary hours** — `backend/internal/google/calendar.go:687`
   - duur is only computed inside the !isAllDay branch, yet all-day events matching schedule keywords are still stored as real 'Dienst' shifts. cron_schedule.go:80 and executor.go:934 sum ev.Duur unconditionally, so an all-day care shift contributes 0h — the owner is reported as under his 16h contract / missing real salary hours, with no warning. This feeds an actual paycheck.
3. 🟠 **Habit log/Toggle/Incident/ForDate default to UTC date while the store uses Europe/Amsterdam** — `backend/internal/handler/habit.go:268 (also :330, :444)`
   - Handlers default to time.Now().Format() in process-local UTC (Render runs UTC); the store uses todayAmsterdam(). Every night between 00:00–02:00 Amsterdam a completion is stamped on yesterday, so RefreshHabitProgress sees today as a miss and the streak breaks/resets. Fires nightly on a core gamified feature, worsened at DST. Centralize one Amsterdam date helper.
4. 🟠 **Streak walk ignores due-days; non-due days break the run** — `backend/internal/store/habit.go:533 (and longestDateRun:507)`
   - currentDateRun/longestDateRun walk strictly consecutive calendar dates and halt on the first gap. For weekenddagen/aangepast/rooster_filter habits with non-consecutive due days, off days (which can never be completed) break the chain, so the owner's shift-filtered habit streaks are systematically undercounted. Skip non-due days via habitDueOnDate; only reset on a missed DUE day.
5. 🟠 **x_per_week / x_per_maand frequencies treated as daily; doel_aantal never evaluated** — `backend/internal/store/habit.go:242`
   - These are real, user-selectable frequencies (ai/tools.go, telegram_dashboards.go), but habitFrequencyDueOnDate returns true every day for them, so completion rate, heatmap, and streaks treat them as daily and reset on any single missed day. doel_aantal is used only for label text, never in due/streak math. Either implement periodic counting against doel_aantal or reject these at create/update — don't ship a feature that silently corrupts its own metric.
6. 🟡 **Note link-sync runs in a post-commit transaction; its failure returns 500 for a write that already succeeded** — `backend/internal/store/note.go:229 (also Create:119, RestoreRevision:493)`
   - tx.Commit at note.go:226 precedes SyncLinksFromContent, which opens its own transaction; a sync error propagates to the handler as HTTP 500 even though the note is saved. The client sees failure for a committed write and a naive retry re-saves. Thread the existing tx in for atomicity, or treat link-sync failure as non-fatal (log + 200). Self-heals on next edit, hence medium.

## Themes
- Schema drift between dead migrations/ and the live runtime bootstrap: the unique indexes that every ON CONFLICT depends on exist only in migrations/002 (never executed), so the runtime schema is silently incomplete and only the legacy prod DB survives by accident. This is a systemic single-source-of-schema-truth problem, not two isolated bugs.
- Timezone boundary inconsistency: the store is rigorously Amsterdam-local (todayAmsterdam) but HTTP handlers default dates to process-local UTC on a UTC server. The layers disagree on what 'today' is, corrupting any midnight-window write. A single shared date helper would close the whole class.
- Silent-failure / wrong-number-not-crash pattern: every high finding fails quietly and produces a plausible result — 0 hours, a broken streak, a daily-treated periodic habit. For a solo dev who trusts these numbers (salary, streaks) without cross-checking, silent corruption is more dangerous than a crash.
- Features accepted at the edge but unimplemented in the core: x_per_week/x_per_maand/doel_aantal and all-day shifts are accepted, persisted, and rendered, but the streak/completion/hours math doesn't honor them. Validation and computation are out of sync.
- Tests assert the happy read path but not the failing write path: FreshDB and runtime-boot tests pass precisely because they never exercise the ON CONFLICT upserts that break — coverage gives false confidence on exactly the DR scenario it was written to guard.

## Notably solid
- The store layer's timezone discipline is genuinely good — todayAmsterdam() consistently localizes refresh/stats/heatmap to Europe/Amsterdam, and the UTC-boundary risk is even explicitly documented in cron_schedule.go:61. The bug is purely the handler layer not following the store's lead, not a missing concept.
- Upserts are correctly written as idempotent ON CONFLICT (user_id, event_id) DO UPDATE with full column refresh — the data-access design is right; only the matching index in the runtime bootstrap is missing. The fix is additive, not a rewrite.
- Schedule writes are properly transactional with defer tx.Rollback, so the critical defect fails safe (clean rollback, no partial/corrupt rows) rather than leaving torn data — the failure is total-but-clean, which makes recovery straightforward once the index is added.
- The backlink graph self-heals: SyncLinksFromContent fully replaces a note's source links on every content edit, so even the medium note-sync inconsistency converges on the next save rather than drifting permanently.
- Habit frequency handling for the calendar-aligned cases (weekdagen, weekenddagen, aangepast day-of-week) is correct and clean; the streak/frequency bugs are confined to the periodic (x_per_*) and non-consecutive due-day cases, not the everyday daily habit.

## All confirmed findings

### 🔴 CRITICAL (1)

**1. BulkUpsert ON CONFLICT(user_id,event_id) has no matching unique index in runtime bootstrap**  
- _Schedule / rooster (diensten, CSV/Calendar import, range queries)_ · `data-integrity` · `backend/internal/store/schedule.go:103`
- **Impact:** On any fresh or restored database (new Render instance, DR rebuild, restore into empty DB, local dev), the schedule table is created by ensureBaseTables WITHOUT a unique index on (user_id, event_id). The very first schedule sync — cronScheduleWeeklyCheck/syncToScheduleStore (cron_workers.go:325), the HTTP sync handler (sync.go:91), the manual import handler (handler/schedule.go:98), and telegram_commands.go:321 — calls BulkUpsert whose ON CONFLICT (user_id, event_id) raises Postgres 42P10 ("no unique or exclusion constraint matching the ON CONFLICT specification"). The whole transaction (tx.Rollback at schedule.go:94) is discarded, so the schedule table never populates. The identical defect affects personal_events: ensureBaseTables creates it without idx_pe_user_event, and PersonalEventStore.UpsertSynced (personal_event.go:158) / BulkUpsertSynced (personal_event.go:179) use the same ON CONFLICT (user_id, event_id). Downstream, the salary/contract weekly check (cron_schedule.go:80) and contract analyse (executor.go:934) then read an empty schedule and silently report 0 planned hours.
- **Fix:** Add CREATE UNIQUE INDEX IF NOT EXISTS idx_schedule_user_event ON schedule(user_id, event_id) and idx_pe_user_event ON personal_events(user_id, event_id) inside ensureBaseTables (mirroring migrations/002:29 and 002:150). Extend TestEnsureRuntimeSchema_FreshDB to actually exercise ScheduleStore.BulkUpsert and PersonalEventStore.UpsertSynced against the fresh schema so the missing constraint is caught at CI time, not in production after a DR rebuild.

### 🟠 HIGH (6)

**1. Streak ignores due-days; non-due days break the run**  
- _Habits gamification (streaks, XP, badges, quantitative, incidents)_ · `correctness` · `backend/internal/store/habit.go:533`
- **Impact:** For habits whose due days are not calendar-consecutive (frequentie weekenddagen/aangepast, or rooster_filter werkdagen/vrijedagen/vroegedienst/latedienst), currentDateRun/longestDateRun walk strictly consecutive calendar dates and stop at the first day with no completion. Off days (which cannot be completed) therefore break the run, so streaks for these habit types are systematically undercounted and reset whenever a non-due day falls between two due days.
- **Fix:** Make currentDateRun/longestDateRun (or calculateHabitProgress) skip non-due days using habitDueOnDate with per-day schedule context, resetting the run only on a missed *due* day.

**2. x_per_week/x_per_maand and doel_aantal accepted but ignored in due/completion logic**  
- _Habits gamification (streaks, XP, badges, quantitative, incidents)_ · `correctness` · `backend/internal/store/habit.go:242`
- **Impact:** x_per_week and x_per_maand are first-class selectable frequencies (exposed in the AI tool schema and rendered as 'Nx per week/maand' in telegram_dashboards.go using doel_aantal), but habitFrequencyDueOnDate returns true for them on every day. They are therefore treated as daily: due every calendar day, so completionForDate/heatmap tank the completion rate, the periodic target (doel_aantal) is never evaluated, and streaks reset whenever any single day is missed. doel_aantal is never consulted anywhere in the streak/completion path.
- **Fix:** Implement periodic per-week/per-month logic against doel_aantal (e.g. count completions within the period vs target) or reject these frequencies at create/update; either wire doel_aantal into due/completion or remove it.

**3. Handlers default log date to UTC not Amsterdam**  
- _Habits gamification (streaks, XP, badges, quantitative, incidents)_ · `dates-timezone` · `backend/internal/handler/habit.go:268`
- **Impact:** Toggle (l.268), Incident (l.330) and ForDate (l.444) default the log date to time.Now().Format("2006-01-02"), which on Render (server TZ = UTC) yields the UTC calendar day. The store deliberately uses todayAmsterdam() for refresh/stats/heatmap. Between 00:00 and ~02:00 Amsterdam (CEST UTC+2 / CET UTC+1) a completion is stamped on the previous calendar day, so RefreshHabitProgress/Stats (which use todayAmsterdam) see no completion for 'today' — today shows as a miss and the current streak breaks at midnight. ForDate also reads/writes the wrong day for the late-night window.
- **Fix:** Use the Amsterdam date helper in all three handlers (and centralize a single shared date helper instead of duplicating time.Now().Format) so handler-defaulted dates match the store's todayAmsterdam().

**4. Unique index personal_events(user_id, event_id) is missing from the runtime schema — every upsert fails on a fresh/DR-restored DB**  
- _Personal events (lifecycle, all-day/multi-day, conflict detection)_ · `data-integrity` · `backend/internal/store/runtime_schema_base.go:187`
- **Impact:** Both Upsert (personal_event.go:158) and upsertSyncedSQL (personal_event.go:179) use ON CONFLICT (user_id, event_id) DO UPDATE. Postgres requires a unique index/constraint on exactly those columns for that clause; without one it errors 'there is no unique or exclusion constraint matching the ON CONFLICT specification'. On the fresh/restore/DR path (EnsureRuntimeSchema only — confirmed the only schema bootstrap; migrations/ is never executed by Go), every personal-event write fails: sync upsert, AI afspraakMaken/afspraakBewerken, and manual create.
- **Fix:** Add CREATE UNIQUE INDEX IF NOT EXISTS idx_pe_user_event ON personal_events(user_id, event_id) to a runtime ensure* step (e.g. inside ensureBaseTables right after the CREATE TABLE, or ensurePersonalEventRetrySchema). Extend TestEnsureRuntimeSchema_FreshDB to perform a real PersonalEventStore.Upsert against the fresh schema so this drift is caught in CI rather than after a DR rebuild.

**5. All-day work shifts get Duur=0 in salary/contract hour totals**  
- _Schedule / rooster (diensten, CSV/Calendar import, range queries)_ · `correctness` · `backend/internal/google/calendar.go:687`
- **Impact:** In parseScheduleEvent, duur is computed (eindDt.Sub(startDt)) only inside the `if !isAllDay` branch (calendar.go:687-689); for an all-day event duur stays at its zero value 0.0. All-day events are NOT filtered out of the schedule: the SDB fetch loop (calendar.go:171-194) only filters by include/exclude keywords on title/description, and getShiftType returns "Dienst" for all-day events (calendar.go:812-813), so an all-day dienst is stored as a normal work shift with duur=0. The weekly contract check sums ev.Duur into totalHours (cron_schedule.go:80) and compares against the hardcoded 16h contract, and executeContractAnalyse buckets ev.Duur per week (executor.go:933-934) — an all-day shift contributes 0 hours, so the owner is reported as under his 16h contract / missing hours.
- **Fix:** In parseScheduleEvent, either reject/flag all-day events that match schedule keywords, or derive a duration for them (e.g. a configured default shift length), and emit a slog.Warn when a stored dienst ends up with Duur==0 so silent zero-hour shifts surface. The contract-check consumers (cron_schedule.go:80, executor.go:934) could also treat heledag shifts explicitly.

**6. Habit log/today handlers use server-UTC date while the store uses Europe/Amsterdam - wrong calendar day after midnight (DST-relevant)**  
- _Cross-cutting: dates/timezones, data-integrity, security, Todoist_ · `dates-timezone` · `backend/internal/handler/habit.go:268`
- **Impact:** Toggle/log (handler/habit.go:266-268), Incident (handler/habit.go:330) and ForDate (handler/habit.go:443-444) default the date with time.Now().Format("2006-01-02") in the process-local zone (UTC on Render), while store/habit.go:644 todayAmsterdam() and RefreshHabitProgress (habit.go:436 -> calculateHabitProgress/currentDateRun) treat 'today' as the Europe/Amsterdam date. - **Fix:** Default the date in Toggle, Incident and ForDate with the same Europe/Amsterdam helper the store uses (todayAmsterdam) instead of time.Now().Format("2006-01-02"), so the HTTP and store layers agree on the day boundary.

### 🟡 MEDIUM (9)

**1. Link-sync runs in a separate post-commit transaction, so its failure returns 500 after the note already saved**  
- _Notes (CRUD, tags, backlinks, revisions, search)_ · `correctness` · `backend/internal/store/note.go:229`
- **Impact:** In update(), tx.Commit happens at note.go:226, then SyncLinksFromContent is called at note.go:230 inside its OWN new transaction (Begin at note.go:590). If link sync errors, update() returns that error (note.go:231) and the handler responds HTTP 500 (handler/note.go:281) even though the note write already committed. Same shape in Create (note.go:119-121 -> handler 144) and RestoreRevision (note.go:493-495). Result: client sees a failure/500 for a write that actually succeeded, and the backlink graph is left out of sync until the next edit; a naive client retry re-saves an already-saved note.
- **Fix:** Thread the existing tx into SyncLinksFromContent so link recomputation commits atomically with the note write; or, if it must stay separate, treat link-sync failure as non-fatal (log + return the saved note with 200) so the response reflects the committed state.

**2. Update concatenates raw JSON keys as SQL identifiers (mass-assignment, no whitelist)**  
- _Habits gamification (streaks, XP, badges, quantitative, incidents)_ · `security` · `backend/internal/store/habit.go:137`
- **Impact:** HabitStore.Update ranges over the decoded JSON map and concatenates each key directly as a SQL identifier (`col+" = $"+i`). The handler (l.119-128) decodes the PATCH body into map[string]any and passes it through with no key whitelist. Any column can be mass-assigned, including gamification-integrity fields (huidige_streak, totaal_xp, totaal_voltooid, langste_streak), ownership (user_id), and is_actief/id; an unknown/malformed key surfaces as a raw 500 SQL error instead of a clean 400. Values are parameterized so value injection is not possible, but identifier handling is unsafe.
- **Fix:** Whitelist patchable columns in Update (or the handler); reject anything outside an allowed set and explicitly exclude huidige_streak, langste_streak, totaal_voltooid, totaal_xp, user_id, id, is_actief.

**3. Badge xp_bonus never aggregated into any XP total**  
- _Habits gamification (streaks, XP, badges, quantitative, incidents)_ · `correctness` · `backend/internal/store/habit.go:451`
- **Impact:** awardHabitBadges writes habit_badges.xp_bonus (10-5000 across streak/total badges) but never adds it to totaal_xp. RefreshHabitProgress computes xp solely as sum(log.XPVerdiend) (l.463) and writes it to habits.totaal_xp (l.443-448); Stats.TotaalXP is SUM(totaal_xp) (l.704). Badge bonuses are therefore excluded from every aggregate XP figure. The frontend only renders xpBonus as a decorative per-badge '+N XP' label (BadgeShowcase.tsx l.103) and never folds it into the displayed total, so the user never actually receives those points in their XP score.
- **Fix:** Either add SUM(habit_badges.xp_bonus) into the authoritative XP total (in RefreshHabitProgress/Stats) or drop XPBonus from the badge model to avoid implying points that are never granted.

**4. Plain Upsert has no optimistic-concurrency guard — a stale client POST silently overwrites a freshly-synced (Confirmed) event**  
- _Personal events (lifecycle, all-day/multi-day, conflict detection)_ · `correctness` · `backend/internal/store/personal_event.go:158`
- **Impact:** The edit modal POSTs the full row it loaded, keyed by event_id. If a background sync updates that row (new time/title from Google) between the modal opening and the user saving, the unguarded plain Upsert overwrites the freshly-synced row with the stale client copy — a classic lost update. There is no updated_at/version check on this path. Single-tenant usage and the owner proxy gate make the race window narrow, so this is correctness/UX rather than critical, but the write is silent and unrecoverable.
- **Fix:** Add an optimistic-concurrency check: carry an updated_at/version column on personal_events and reject/no-op the Upsert when the client's base version is older than the stored row, or route the edit path through a guarded update. At minimum return a 409 so the client re-fetches instead of clobbering.

**5. weeknr() pairs calendar Year() with ISO week, producing wrong year-week at year boundaries**  
- _Schedule / rooster (diensten, CSV/Calendar import, range queries)_ · `dates-timezone` · `backend/internal/google/calendar.go:847`
- **Impact:** weeknr (calendar.go:847-850) does `_, week := d.ISOWeek(); return fmt.Sprintf("%d-W%02d", d.Year(), week)`. It pairs the Gregorian calendar year (d.Year()) with the ISO week number, but ISO week 1 and week 52/53 can belong to a different ISO year than the calendar year at the Dec/Jan boundary. This weeknr is stored on each dienst and used as the grouping key in executeContractAnalyse (executor.go:933: weekMap[ev.Weeknr]). Shifts in a boundary week get mislabeled (and split across two different keys) so per-week hours/deltas for those weeks are wrong.
- **Fix:** Use isoYear, week := d.ISOWeek() and format with isoYear instead of d.Year(). A schedule resync rewrites weeknr for affected rows. Note: the example in the original finding (2026-12-29 -> 2026-W01) is incorrect — Dec 29 2026 is genuinely W53; the real failing cases are e.g. 2025-12-29 (code yields 2025-W01, correct is 2026-W01) and 2027-01-01 (code yields 2027-W53, correct is 2026-W53).

**6. Habit Update interpolates untrusted JSON object keys into the SQL SET clause (SQL injection / no column allowlist)**  
- _Cross-cutting: dates/timezones, data-integrity, security, Todoist_ · `security` · `backend/internal/store/habit.go:137`
- **Impact:** HabitStore.Update (habit.go:133-147) builds the SET clause as col+" = $"+i directly from the keys of the map passed in. HabitHandler.Update (handler/habit.go:119) decodes the request body straight into map[string]any and forwards it unfiltered, so every JSON object key becomes raw SQL column text (column names cannot be parameterized by pgx). Note: the recommendation's reference to automation.go:108 is inaccurate — that file is 99 lines and uses a fixed-column UPDATE; the real sibling pattern is store/note.go (already mitigated at the handler layer).

**4. Editing a PendingDelete event resurrects it as PendingUpdate**  
- _Personal events (lifecycle, all-day/multi-day, conflict detection)_ · `bug` · `backend/internal/engine/executor.go:2293`
- **Impact:** If an event is queued for deletion (PendingDelete) and is then edited (AI tool or edit modal) before the delete is pushed to Google, its status flips to PendingUpdate and the queued delete is silently cancelled — the event is resurrected. There is no transition table preventing PendingDelete→PendingUpdate. Low likelihood because the UI generally hides events queued for deletion, but the lifecycle permits it.
- **Fix:** Introduce an explicit allowed-transition check (refuse to edit a row in PendingDelete, or require re-create) so a queued delete cannot be silently turned into an update. Centralize the transition rules rather than re-deriving the next status independently in the frontend and the executor.

**6. Writes trust user_id from the request body instead of an authenticated identity (latent IDOR if multi-user)**

**7. Todoist due-date slice and today fallback are unguarded (panic / silently-wrong window)**

**8. Habit progress recompute is a non-transactional read-modify-write on langste_streak**  
- _Cross-cutting: dates/timezones, data-integrity, security, Todoist_ · `concurrency` · `backend/internal/store/habit.go:436`
- **Impact:** RefreshHabitProgress (store/habit.go:408-451) runs s.Get (line 409), s.db.Pool.Query for all logs (line 413), and the UPDATE (line 443) as three separate pool operations with no transaction or SELECT ... FOR UPDATE. calculateHabitProgress then floors longest with the previously-read habit.LangsteStreak (lines 436-439). Two concurrent log writes for the same habit (fast double-tap, or a web toggle racing the cron) can both read the same stale langste_streak and the later UPDATE wins. Because each side fully recomputes from the log set, the window is small and self-healing on the next write, so impact is minor for a single-user app.
- **Fix:** Wrap the read-recompute-write in a transaction with SELECT ... FOR UPDATE on the habit row, or compute langste_streak with GREATEST(langste_streak, $new) in the UPDATE so a concurrent writer cannot lower it.
